Investigation report

Claude Code agents & dynamic workflows, adapted for Pi

How Claude Code structures subagents, background sessions, and script-driven workflows in its TUI — and a concrete, phased plan for bringing the best of that design to the existing Herdr-backed Pi subagent runtime.

Sources: official Claude Code documentation (code.claude.com, CLI v2.1.212), Ben Davis' my-pi-setup (commit d8534d7e), and the local amp-subagents extension. No repository files were modified.

1Executive conclusion

Claude Code's key architectural idea is not any single agent feature. It is the deliberate separation of parallel work into four distinct products, each with its own coordinator, lifecycle, and UI surface:

Claude Code's four coordination primitives
PrimitiveCoordinatorBest useUX surface
SubagentParent model, turn by turnFocused, context-isolated side taskInline panel, @agent, /tasks
Background sessionUserIndependent, durable taskFull-screen claude agents dashboard
Agent teamLead agent + shared task listA few collaborating peersInline agent panel or split panes
Dynamic workflowSandboxed scriptLarge, phased, repeatable fan-outPhase-oriented /workflows dashboard
Recommendation in one paragraph

Preserve the existing durable, Herdr-backed amp-subagents runtime. Build a native /agents attention dashboard over its persistent state. Harden concurrency, lifecycle, watchdogs, and result delivery, and add schema-validated child outputs. Then implement dynamic workflows as a separate sandboxed extension that shares lower-level child infrastructure — without creating a Herdr tab for every workflow child. Defer agent teams.

2Claude's subagent model

Agent definitions

Custom agents are Markdown files with YAML frontmatter and a Markdown system prompt:

---
name: code-reviewer
description: Review code after changes
tools: Read, Grep, Glob
disallowedTools: Write, Edit
model: sonnet
effort: high
background: true
isolation: worktree
maxTurns: 20
---

Review for correctness, security, and maintainability.

Supported policy covers tool allow/deny lists, model and effort, permission mode, max turns, preloaded skills, scoped MCP servers, lifecycle hooks, persistent memory, background policy, worktree isolation, display color, and an initial prompt when the definition runs as a main-session agent.

Definition precedence (highest first): managed settings → session-only --agents JSON → project .claude/agents/ → user ~/.claude/agents/ → plugin definitions. Built-ins include the read-only Explore and Plan agents plus a general-purpose agent.

Notable design shift

As of v2.1.198, /agents no longer opens a creation wizard. Users ask Claude to create an agent, or edit the Markdown directly. Configuration became conversational while the durable file stays reviewable — a pattern worth copying.

Invocation

  • Natural language — Claude decides whether to delegate, based on the request, context, and the agent's description.
  • @agent mention — guarantees that agent runs for this task. The parent still writes the actual delegation prompt.
  • claude --agent <name> — the definition governs the whole main session.

Context isolation

A normal subagent receives its own system prompt, environment details, the delegation prompt, applicable project instructions and memory, explicitly preloaded skills, and its own tools and permissions. It does not inherit parent conversation history, files the parent already read, previously invoked skills, or parent tool transcripts. Only the final summary crosses back into the caller's context.

Claude also supports a distinct fork primitive that inherits the full conversation, tools, model, history, and prompt cache — the inverse trade-off. It should not be conflated with fresh-context delegation.

Nested delegation

Claude allows nesting to depth five, removing the Agent tool at the limit. Only the top-level child's summary reaches the main conversation. For Pi, ordinary children should stay unable to delegate by default: a depth limit alone does not control aggregate cost, concurrency, or lifecycle. Workflows should be the explicit fan-out mechanism.

3Foreground & background execution

Since v2.1.198, subagents run in the background by default; Claude uses foreground execution only when it cannot continue without the result.

Execution modes
AspectForegroundBackground
Parent conversationBlockedRemains usable; children run concurrently
Result deliveryDirect, same turnLater completion notification
Permission promptsPassed through immediatelySurface in main session, naming the child; Esc denies one call without stopping the agent
VisibilityInline/tasks, including finished agents
ControlsCtrl+B to backgroundCtrl+X Ctrl+K (confirmed twice) stops all

Completed named agents can be resumed with follow-up messages, keeping their full history and agent ID. Subagent transcripts persist separately and survive parent compaction and session resumption. Explore and Plan remain one-shot with no resumable ID.

4In-session agents UX

Claude renders a compact agent panel below the prompt. Nested rows show descendant counts such as (+N); opening an agent shows its transcript, siblings, direct children, and a path back to main.

Agent panel controls
KeyAction
↑ / ↓Select an agent row
EnterOpen the agent's transcript and send follow-ups
xStop a running agent, or dismiss a finished one
EscReturn focus to the main prompt

While viewing a child, plain messages and skills target that child; built-in commands still execute in the main session. Named agents appear in @ completion with status. /tasks manages background shells and subagents and is distinct from Claude's internal task checklist (Ctrl+T).

Transferable principle: progressive disclosure

Compact status row → peek / current question → child detail → full transcript or attached session. Pi currently jumps from status text straight to a full Herdr terminal; the missing middle layer is the biggest UX gap.

5Agent view & background sessions

claude agents is a full-screen supervisor for complete, durable sessions — not in-session subagents. Rows are grouped by attention: Pinned, Ready for review, Needs input, Working, Completed. Each row shows a state-colored or animated icon, the session name, a model-generated semantic activity headline, elapsed time, and pull-request info where relevant.

Core interaction

Agent view keyboard controls

SpacePeek at a session without leaving the dashboard

←Detach back to the dashboard (never cancels)

Peek adapts to state: a waiting session shows the exact question with a reply input; a working session shows its status sentence; a completed one shows its result; a PR-linked one shows PR status. Background sessions automatically isolate edits in worktrees and can commit, push, and open draft PRs under constrained rules.

Two UX principles worth copying

State over raw logs — rows show a short semantic headline, not the latest tool invocation; raw activity is available deeper in the hierarchy. Detach ≠ cancel — leaving a child view never stops it; navigation and cancellation are separate actions. The dashboard exists to answer: what needs me, what is progressing, what finished, and where is the result.

6Permissions, models, hooks, worktrees

Tool restrictions

  • tools is an allowlist; omission inherits the parent's available tools.
  • disallowedTools removes tools from the inherited or specified set; denials apply before allowlist resolution.
  • MCP server-wide patterns (mcp__server, mcp__server__*, mcp__*) are supported.
  • Specific agents can be denied via permissions.deny: ["Agent(name)"]; denying bare Agent disables all delegation.
  • Permission precedence is harness-enforced: deny → ask → allow. Agent messages can never approve another agent's permission request or stand in for user consent.

Model resolution

Child model resolves in order: CLAUDE_CODE_SUBAGENT_MODEL env override → per-invocation parameter → agent definition → parent model. Organization model restrictions still apply. Children inherit the parent's extended-thinking state; definitions can set effort but there is no separate thinking toggle.

Hooks

Relevant lifecycle events: SubagentStart, SubagentStop, PreToolUse, PostToolUse, PermissionRequest, TaskCreated, TaskCompleted, WorktreeCreate, WorktreeRemove. Agent-frontmatter hooks exist only while that agent runs; a local Stop hook becomes SubagentStop when delegated. Policy lives in the runtime, not in prompt obedience.

Worktrees

  • isolation: worktree gives a subagent a temporary checkout, branched by default from refreshed origin/HEAD (configurable to current HEAD).
  • Unchanged worktrees are removed automatically; changed ones are retained.
  • .worktreeinclude copies selected gitignored files into fresh worktrees.
  • WorktreeCreate/WorktreeRemove hooks can replace git behavior for other VCS systems.
  • Background sessions auto-isolate before their first in-repo edit unless disabled; children inherit the session's worktree unless separately isolated.

The local Pi implementation's explicit apply / retain / discard worktree actions are safer than Claude's auto-cleanup and should be preserved.

7Dynamic workflows

A dynamic workflow is model-authored JavaScript executed by a dedicated runtime. The script — not the parent model — owns the control flow. Claude writes the script for the task you describe; the runtime executes it in the background while the session stays responsive.

export const meta = {
  name: "audit-routes",
  description: "Audit route authentication",
};

const discovery = await agent("List every route file.", {
  schema: {
    type: "object",
    required: ["files"],
    properties: {
      files: { type: "array", items: { type: "string" } },
    },
  },
});

const audits = await pipeline(
  discovery.files,
  file => agent(`Audit ${file}`, { label: file }),
);

return audits.filter(Boolean);

Why code instead of prose orchestration

The script owns loops, fan-out, branching, phase transitions, intermediate values, aggregation, and retries. Intermediate results stay in script variables instead of expanding the parent context; only the final aggregate returns. This buys larger fan-out, repeatability, typed intermediate results, explicit failure handling, cross-agent verification, and low parent-context pressure.

Runtime constraints (documented)

  • Up to 16 concurrent agents, fewer on constrained machines
  • Up to 1,000 agent calls per run
  • No direct shell or filesystem access from orchestration code — agents do the actual work
  • No general mid-run user input; only agent permission prompts can pause a run
  • Paused runs cache completed agent results; agents interrupted mid-flight restart on resume
  • Resume works only within the same Claude session

Invocation and approval

Workflows start via /deep-research, saved workflow commands, "use a workflow" phrasing, the ultracode keyword, or session-wide /effort ultracode. Before running generated code, Claude shows the planned phases with options: Yes, run it · Yes, and don't ask again · View raw script · No. Ctrl+G opens the script in an editor; Tab lets you adjust the prompt first.

Progress UX

A compact progress line appears below the editor; /workflows opens the full hierarchy:

Workflow run
└── Phase        (agent counts, tokens, elapsed)
    └── Agent
        ├── Prompt
        ├── Recent tool activity
        └── Result

Controls: ↑/↓ select · Enter/→ drill in · Esc/← back · f filter by status · p pause/resume · x stop · r restart agent · s save as a reusable command.

8Comparison with the Pi implementation

The current implementation lives at common/.pi/agent/extensions/amp-subagents/ and is already stronger than basic Claude subagents in durability, direct observability, and mutation safety:

  • Built-in, user, and project agent definitions with trust-gated project agents
  • Model, thinking, tool, and permission policies
  • Background-first execution with persistent disk-backed jobs
  • Interactive Herdr child terminals with parent restart/reload resilience
  • Structured activity snapshots, follow-up messaging, resumption
  • Permission forwarding; worktree-isolated mutation with apply/retain/discard
  • Notification leasing; focus, close, cancel, and cleanup actions

Gaps

Current gaps against the Claude Code design
GapDetail
No native dashboard/agents emits formatted notifications; it lacks state grouping, filtering, peek, in-place reply, permission views, transcript navigation, worktree review controls, and keyboard-driven interaction.
No workflow primitiveNo script-owned abstraction for phases, dynamic fan-out, typed intermediate values, aggregation, or run-level budgets.
No race-safe global concurrencyCapacity is not reserved synchronously before asynchronous launch work, so parallel tool calls could exceed intended limits.
No enforced structured outputreturnMode is advisory metadata, not a schema-backed contract.
Result consumption unclearExplicit result retrieval and automatic completion injection need a shared consumed/obsolete state to prevent duplicate parent-context delivery.
No workspace validationamp-subagents is not included in the aggregate common/.pi/package.json check/test/typecheck scripts.

9Lessons from Ben Davis' setup

After the independent Claude investigation, artifacts/ben-subagents-findings.md and the referenced source were reviewed. Ben's implementation independently reaches the same core boundary Claude drew: ordinary delegation ≠ workflow orchestration. Its workflow path:

Model-authored JavaScript
        │
        ▼
Permission-restricted Node subprocess
        │  narrow, bounded, token-authenticated IPC
        ▼
Workflow controller (semaphore, budgets, settle)
        │
        ▼
Isolated child runners

Strong properties worth adopting:

  • phase(), agent(), and parallel() primitives; scripts orchestrate, agents act
  • Global concurrency of four; maximum 32 child calls per run
  • No filesystem, network, process, imports, eval, or timers in orchestration code
  • Bounded source (512 KB), args (256 KB), IPC messages, and result (1 MB) sizes
  • { ok, output, structured, error } results — agent() never throws into the script
  • Schema-backed terminating structured_output tool
  • First-response watchdog (45 s) and per-tool timeout (3 min); no overall run deadline
  • Persisted scripts, progress, transcripts, and results; native /workflows dashboard

This is a better basis for Pi workflows than letting ordinary children recursively invoke the existing Agent tool.

10Recommended Pi architecture

Keep three domain layers

Agent definition   →  reusable role and policy
Agent runtime      →  one delegated durable job
Workflow runtime   →  script-owned orchestration of many children

They may share presentation conventions and infrastructure, but must retain separate identity and lifecycle semantics.

Share infrastructure, not lifecycle

Reuse or extract: agent discovery, project trust checks, tool-policy normalization, model resolution, permission guards, usage/transcript normalization, activity summaries, structured-output support, concurrency accounting, and worktree helpers. The workflow extension should depend on a lower-level child-runner interface — not on the public model-facing Agent tool.

Decouple execution from presentation

For deliberate standalone jobs, keep the current semantics: one Agent job → one observable Herdr child. For workflow-scale fan-out, use compact dashboard-managed children with persisted transcripts, promoting a child into a full Herdr terminal only when the user wants direct interaction. A workflow that creates 20 agents must not produce 20 terminal tabs.

Suggested module layout

common/.pi/agent/extensions/
├── agent-core/            # shared services, extracted on demand
│   ├── discovery.ts
│   ├── policy.ts
│   ├── concurrency.ts
│   ├── activity.ts
│   ├── structured-output.ts
│   └── child-runner.ts
├── amp-subagents/         # existing runtime + new dashboard
│   ├── runtime.ts
│   ├── dashboard.ts
│   └── ...
└── workflows/             # new, separate extension
    ├── index.ts
    ├── controller.ts
    ├── sandbox.ts
    ├── sandbox-child.cjs
    ├── persistence.ts
    ├── dashboard.ts
    └── model.ts

Do not perform a broad framework extraction up front; extract each shared unit only when the second consumer actually needs it.

11Implementation sequence

Phase 1

Harden amp-subagents

  1. Add a package manifest; include tests and typecheck in common/.pi/package.json.
  2. Add a process-wide race-safe semaphore; reserve capacity synchronously before async launch work; release exactly once on every terminal path.
  3. Add a first-assistant-event watchdog (45–60 s) and per-tool timeouts (~3 min) without a whole-task deadline.
  4. Correct child extension startup and bounded, idempotent shutdown.
  5. Separate durable history from result-delivery state; add consumed/obsolete states.
  6. Add schema-backed child results.
Phase 2

Native /agents dashboard

Group jobs by Needs input · Working · Completed. Rows show state, agent and job ID, semantic activity, backend, elapsed time, model/effort, and worktree state. Suggested controls: Space peek · Enter detail · r reply/resume · a/d approve/deny · f focus Herdr child · x cancel · A/R/D apply/retain/discard · / filter · Esc back. The dashboard is the durable control plane; Herdr stays the deep interactive terminal.

Phase 3

Workflow MVP

A separate workflow tool exposing phase(title), agent(prompt, { label?, agentType?, schema?, model?, effort? }), and parallel(thunks, { concurrency? }). Initial limits: 4 concurrent children · 32 calls · 512 KB source · 256 KB args · 1 MB result · first response within 45–60 s · 3-minute per-tool timeout · no overall deadline. Persist under:

~/.local/state/pi/workflows/<run-id>/
├── script.js
├── args.json
├── state.json
├── result.json
└── agents/<agent-id>/{result.json, transcript.json}
Phase 4

/workflows dashboard

Run list → phase list → child list → child detail; cancel run or child; restart failed child; inspect persisted output. Add pause/resume only after cancellation, persistence, and restart semantics are reliable.

Phase 5

Conversational workflow creation

Explicit "use a workflow" prompting; approval showing phases and raw script with external-editor review; user- and project-scoped saving; structured arguments; optional size guidance; a /workflow or /ultracode skill. Establish explicit, inspectable behavior before any magical trigger keyword.

12What not to copy

Anti-patterns
  • Do not build agent teams yet. They require shared task mutation, mailboxes, peer discovery, direct teammate interaction, conflict handling, and harder resumption. Claude itself still marks them experimental.
  • Do not merge all background concepts. Shell jobs, subagent jobs, background sessions, and workflow runs may share visual language but not lifecycle rules.
  • Do not execute generated code in the Pi process. Model-authored JavaScript belongs in a permission-restricted, killable subprocess with narrow authenticated IPC and strict payload limits.
  • Do not expose raw logs as primary status. Persist raw activity, but show semantic summaries in list rows.
  • Do not permit unrestricted recursive delegation. Workflows are the bounded, inspectable fan-out mechanism.
  • Do not create a Herdr tab per workflow child. That works for a few deliberate jobs, not workflow-scale orchestration.

13Final recommendation

Pi conversation
├── Agent      →  one focused durable delegation (Herdr-backed)
├── /agents    →  user-facing job supervisor (attention dashboard)
└── Workflow   →  sandboxed, phased orchestration
    └── /workflows  →  run and phase supervisor

Preserve the current Herdr-backed runtime for individually important jobs. Add the missing dashboard layer. Harden the child primitive before using it as shared infrastructure. Then introduce workflows as a separate bounded orchestration runtime, inspired by Claude's semantics and Ben's sandbox/controller design.

Guiding principle

Compact by default, inspectable at every level, durable across lifecycle changes, and explicit about whether coordination belongs to the user, the parent model, a lead agent, or a workflow script.
